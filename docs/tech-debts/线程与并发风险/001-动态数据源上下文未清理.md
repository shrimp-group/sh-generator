## 描述
`GenService.getTables` / `getColumns` 在读取目标库元数据前调用 `DynamicDataSourceHolder.set(project.getDbCode())` 切换数据源，但生成完成后未显式调用 `clear` / `remove`：

```java
private List<TableInfo> getTables(GenProject project, GenDatasource datasource) {
    ...
    DynamicDataSourceHolder.set(project.getDbCode());
    return tableInfoMapper.getTables(info);
}
```

`DynamicDataSourceHolder` 通常基于 `ThreadLocal` 实现。在 Tomcat 等使用线程池复用线程的容器中：
1. 当前请求结束后线程被归还线程池，`ThreadLocal` 中的 `dbCode` 残留。
2. 下一个复用该线程的请求若未显式 set，会误连到上一个请求的目标库，导致读取错误的业务库元数据或写入错误数据。

## 影响范围
- 所有使用线程池的部署环境（Tomcat 默认即线程池）。
- 多用户并发生成时可能交叉读取错误数据源。

## 复现条件
1. 用户 A 触发生成（dbCode=dbA）。
2. 紧接着用户 B 触发其他非生成请求，复用同一线程。
3. 用户 B 的请求若走到依赖默认数据源的逻辑，可能误连 dbA。

## 当前解决方案（如有）
依赖框架在请求拦截器中统一 clear，或下次 set 覆盖。本仓库代码未显式 clear。

## 建议解决方案
- 在 `getGenZip` / `getGenData` 的 finally 块中显式 `DynamicDataSourceHolder.clear()`。
- 或在 AOP 切面/拦截器中确保每次生成请求结束后清理 ThreadLocal。
- 单元测试覆盖线程复用场景。

## 关联模块
- `generator-server/src/main/java/com/wkclz/generator/server/service/GenService.java`（getTables、getColumns 方法）
- 依赖 `sh-dynamicdb` 的 `DynamicDataSourceHolder`
